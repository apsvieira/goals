async function handleLogout() {

    syncManager.stopAutoSync();

    // Unregister push notifications
    await unregisterPushNotifications();

    // Clear local data (IndexedDB: goals, completions, meta, operations)

## Task 2: Fix `UpsertCompletion` ON CONFLICT clause (Review #5) — HIGH

**Problem:** `UpsertCompletion` uses `ON CONFLICT(id)`, but new completions generated by sync get fresh UUIDs. If a race or retry creates a second UUID for the same `(goal_id, date)`, the INSERT violates the `UNIQUE(goal_id, date)` constraint instead of upserting. `CreateCompletion` correctly uses `ON CONFLICT(goal_id, date)`.

- Modify: `backend/internal/db/queries.go:794-807` (SQLite)
- Modify: `backend/internal/db/postgres.go:898-917` (Postgres)

func (d *SQLiteDB) UpsertCompletion(c *models.Completion) error {
	now := time.Now().UTC()
	if c.UpdatedAt.IsZero() {
		c.UpdatedAt = now
	}

	_, err := d.Exec(`
		INSERT INTO completions (id, goal_id, date, created_at, updated_at, deleted_at)
		VALUES (?, ?, ?, ?, ?, ?)

			id = excluded.id,
			updated_at = excluded.updated_at,
			deleted_at = excluded.deleted_at

		return fmt.Errorf("upsert completion: %w", err)

## Task 3: Serialize sync operations per user (Review #4) — HIGH

**Problem:** `ApplyChanges` performs multiple reads and writes with no transaction or lock. Concurrent sync requests from different devices for the same user can interleave, causing lost updates.

**Approach:** Add a per-user mutex in the sync service. This serializes all sync operations for a given user with minimal code change. It's effective for single-instance deployments (which this app uses — SQLite/Fly.io single machine).

> **Future scaling note:** For multi-instance Postgres deployments, replace this with advisory locks (`SELECT pg_advisory_xact_lock(hash)`) or wrap ApplyChanges in a DB transaction via a `RunInTransaction(func(Database) error) error` interface method. The mutex approach is correct for the current architecture.

	// Create user

		Color:     "#000000",

		UpdatedAt: time.Now().UTC().Add(-time.Hour),

	var wg sync.WaitGroup
	errs := make([]error, 2)

					},
				},
			}
			_, errs[idx] = svc.ApplyChanges(userID, req)

func (s *Service) getUserLock(userID string) *sync.Mutex {
	s.mu.Lock()

	return s.locks[userID]

## Task 5: Add HTTP server timeouts (Review #7) — HIGH

**Problem:** `http.Server` has no `ReadHeaderTimeout`, `ReadTimeout`, or `WriteTimeout`. A slow client (or attacker) can hold connections open indefinitely (slowloris attack).

**Files:**
- Modify: `backend/cmd/server/main.go:72-75`
- Modify: `backend/internal/api/api_test.go` (verify existing tests still pass)

### Step 1: Add timeouts to http.Server

In `backend/cmd/server/main.go:72-75`, change:

```go
server := &http.Server{
    Addr:    serverAddr,
    Handler: handler,
}
```

to:

```go
server := &http.Server{
    Addr:              serverAddr,
    Handler:           handler,
    ReadHeaderTimeout: 10 * time.Second,
    ReadTimeout:       30 * time.Second,
    WriteTimeout:      60 * time.Second,
    IdleTimeout:       120 * time.Second,
}
```

Rationale for values:
- `ReadHeaderTimeout: 10s` — enough for any legitimate client to send headers; blocks slowloris
- `ReadTimeout: 30s` — enough for large sync request bodies over slow mobile connections
- `WriteTimeout: 60s` — enough for server to process and respond to sync (which may take time for large payloads)
- `IdleTimeout: 120s` — keep-alive connection reuse window
